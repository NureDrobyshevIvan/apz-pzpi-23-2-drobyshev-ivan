3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про шаблони (патерни) проєктування

     Потреба у використанні патернів проєктування виникає через необхідність стандартизації рішень у розробці програмного забезпечення. Під час написання коду без використання загальноприйнятих стандартів розробники часто стикаються з проблемою повторного вирішення вже відомих задач. Такий підхід робить архітектуру заплутаною, ускладнює подальшу підтримку та масштабування системи. Патерни проєктування виступають у ролі концептуальних креслень — перевірених часом рішень для типових архітектурних проблем. Це не готова бібліотека чи фрагмент коду, який можна скопіювати, а вищий рівень абстракції, що допомагає правильно організувати взаємодію класів. Крім того, використання шаблонів формує спільний професійний словник: згадка конкретного патерна одразу пояснює колегам закладений архітектурний підхід.
     Для систематизації підходів до проєктування автори класичної праці «Прийоми об'єктно-орієнтованого проєктування» (відомі як «Банда чотирьох» або GoF) розділили всі двадцять три базові патерни на три основні групи:
     • Породжувальні шаблони (Creational patterns): фокусуються на процесі створення об'єктів. Вони забезпечують незалежність системи від способу ініціалізації її компонентів та приховують складну логіку їхнього конструювання.
     • Структурні шаблони (Structural patterns): відповідають за компонування об'єктів і класів у більші, але гнучкі структури, забезпечуючи ефективну підтримку зв'язків між елементами.
     • Поведінкові шаблони (Behavioral patterns): визначають алгоритми взаємодії, а також способи ефективної комунікації та чіткого розподілу обов'язків між об'єктами.
     Область застосування патернів проєктування охоплює практично всі етапи створення сучасного об'єктно-орієнтованого програмного забезпечення, незалежно від обраної мови програмування. Вони є фундаментальною основою для розробки потужних фреймворків, стандартних бібліотек та архітектурних рішень корпоративного рівня. Патерни застосовуються в тих частинах системи, де вимагається висока гнучкість, здатність до розширення функціоналу в майбутньому та надійна ізоляція складної внутрішньої логіки від клієнтського коду.

3.2 Опис обраного патерна

     Назва патерна: «Будівельник» (Builder)
     Тип шаблона (патерна): Породжувальний (Creational)
     Основне призначення: Патерн вирішує проблему створення складних об'єктів, ініціалізація яких вимагає виконання багатьох кроків або налаштування великої кількості опціональних параметрів. Він дозволяє створювати різні конфігурації об'єкта, використовуючи один і той самий процес конструювання, та ефективно усуває проблему «телескопічного конструктора» (коли конструктор класу перевантажений параметрами).
     Передумови застосування: Використання патерна є доцільним у таких випадках:
     • Конструктор об'єкта приймає занадто багато параметрів, більшість з яких можуть мати значення null або значення за замовчуванням.
     • Процес створення об'єкта вимагає покрокового складання або суворого дотримання певної послідовності ініціалізації полів.
     • Існує потреба створювати різні представлення або конфігурації одного й того ж продукту.
     • Необхідно ізолювати складний код конструювання об'єкта від його основної бізнес-логіки.
     Переваги:
     • Забезпечує повний контроль над процесом конструювання, дозволяючи створювати об'єкти покроково.
     • Ізолює складний код збирання (ініціалізації) від бізнес-логіки класу, дотримуючись принципу єдиної відповідальності (Single Responsibility Principle).
     • Значно покращує читабельність та зрозумілість клієнтського коду, особливо при використанні підходу ланцюжка методів (Fluent API).
     • Дозволяє повторно використовувати один і той самий код будівельника для створення різних конфігурацій продукту.
     Недоліки:
     • Збільшує загальну складність архітектури та обсяг коду за рахунок необхідності створення додаткових класів (інтерфейсів, конкретних будівельників, директорів).
     • Є надлишковим та недоцільним для використання при конструюванні простих об'єктів із малою кількістю полів.

3.3 Структура шаблона (патерна)

     Опис структурних частин: Базова структура патерна «Будівельник», представлена на діаграмі(див. рис. 3.1), складається з чітко розділених компонентів, кожен з яких виконує свою роль у процесі створення складного об'єкта:
     • IBuilder (Інтерфейс Будівельника): абстрактний інтерфейс, який визначає загальний контракт для конструювання. Він декларує метод build(), призначений для повернення готового продукту, а також може містити методи для покрокового налаштування.
     • ObjectBuilder (Конкретний Будівельник): клас, що безпосередньо реалізує інтерфейс IBuilder. Він інкапсулює всю технічну логіку створення продукту, зберігає його проміжний стан під час покрокового складання та формує кінцевий результат.
     • TargetObject (Продукт / Цільовий об'єкт): складний об'єкт, який є кінцевою метою процесу конструювання. Його ініціалізація вимагає виконання багатьох кроків.
     • OtherObject (Складові частини): дрібніші об'єкти, модулі, залежності або параметри, з яких структурно складається фінальний цільовий об'єкт.
     Опис взаємодії об’єктів: Взаємодія компонентів патерна базується на принципі делегування відповідальності за створення об'єкта. Клієнтський код створює екземпляр конкретного ObjectBuilder. Замість прямого створення TargetObject через його власний конструктор, клієнт викликає конфігураційні методи будівельника. Під час цього процесу ObjectBuilder самостійно створює або отримує потрібні екземпляри складових частин (OtherObject) та інтегрує їх у структуру майбутнього продукту. Коли всі необхідні етапи налаштування завершено, викликається фінальний метод build(). Цей метод завершує процес складання і повертає клієнту повністю ініціалізований TargetObject, готовий до безпечного використання в системі.


Рисунок 3.1 – Схема структури патерна

     Наведена UML-діаграма класів наочно візуалізує статичну архітектуру та відносини між елементами шаблону. Зв'язок у вигляді стрілки з порожнім трикутником (позначений як Extends) вказує на відношення реалізації: клас ObjectBuilder зобов'язаний виконати контракт абстрактного IBuilder.
     Спрямована суцільна стрілка з написом Build від конкретного будівельника до TargetObject демонструє відношення створення (залежність) - саме будівельник контролює життєвий цикл продукту на етапі його ініціалізації. Стрілки, що відходять від TargetObject до блоків OtherObject, ілюструють структурну залежність (агрегацію або композицію), вказуючи на те, що цільовий об'єкт виступає контейнером або власником цих дрібніших компонентів.

3.4 Реалізація шаблона (патерна)

     У рамках практичного завдання розроблено програму мовою C#, яка демонструє застосування патерна «Будівельник». За основу взято механізм генерації SQL-запитів. Зазвичай формування складного запиту вимагає конкатенації багатьох рядків з урахуванням різних умов. Будівельник дозволяє вирішити цю задачу елегантно, розбивши процес на окремі, незалежні кроки.
     Спочатку необхідно визначити саму структуру продукту та інтерфейс для його створення.
     1  ﻿// 1. Продукт — складний об'єкт, який ми будуємо
     2  public class SqlQuery
     3  {
     4      public string Table { get; set; } = "";
     5      public List<string> Columns { get; set; } = new List<string>();
     6      public string WhereClause { get; set; } = "";
     7      public string OrderByClause { get; set; } = "";
     8      public override string ToString()
     9      {
    10          var cols = Columns.Any() ? string.Join(", ", Columns) : "*";
    11          var query = $"SELECT {cols} FROM {Table}";
    12
    13          if (!string.IsNullOrEmpty(WhereClause))
    14              query += $" WHERE {WhereClause}";
    15
    16          if (!string.IsNullOrEmpty(OrderByClause))
    17              query += $" ORDER BY {OrderByClause}";
    18
    19          return query + ";";
    20      }
    21  }
    22  // 2. Інтерфейс Будівельника
    23  public interface ISqlQueryBuilder
    24  {
    25      ISqlQueryBuilder SelectColumns(params string[] columns);
    26      ISqlQueryBuilder FromTable(string table);
    27      ISqlQueryBuilder Where(string condition);
    28      ISqlQueryBuilder OrderBy(string column);
    29      SqlQuery Build();
    30  }

     У цьому блоці клас SqlQuery є кінцевим результатом (продуктом). Він містить лише поля для збереження даних та один метод ToString(), який збирає ці дані у фінальний SQL-рядок.
     Щоб не створювати цей об'єкт безпосередньо, ми вводимо інтерфейс ISqlQueryBuilder. Він описує всі можливі дії: вибір таблиці, додавання умов фільтрації чи сортування. Головна особливість цього інтерфейсу в тому, що кожен метод повертає посилання на сам будівельник. Це зроблено для того, щоб у майбутньому ми могли викликати методи один за одним в один рядок (так званий Fluent API).
     Далі реалізується клас, який виконує фактичну роботу з конфігурації об'єкта.
    32  // 3. Конкретний Будівельник
    33  public class MySqlQueryBuilder : ISqlQueryBuilder
    34  {
    35      private SqlQuery _query = new SqlQuery();
    36      public ISqlQueryBuilder SelectColumns(params string[] columns)
    37      {
    38          _query.Columns.AddRange(columns);
    39          return this; // Повертаємо current object для ланцюжка методів
    40      }
    41      public ISqlQueryBuilder FromTable(string table)
    42      {
    43          _query.Table = table;
    44          return this;
    45      }
    46      public ISqlQueryBuilder Where(string condition)
    47      {
    48          _query.WhereClause = condition;
    49          return this;
    50      }
    51      public ISqlQueryBuilder OrderBy(string column)
    52      {
    53          _query.OrderByClause = column;
    54          return this;
    55      }
    56      public SqlQuery Build()
    57      {
    58          var result = _query;
    59          _query = new SqlQuery(); // Скидання для наступного використання
    60          return result;
    61      }
    62  }
     Клас MySqlQueryBuilder інкапсулює всередині себе екземпляр продукту (_query). Коли клієнт викликає будь-який метод налаштування, будівельник просто оновлює відповідне поле цього внутрішнього об'єкта і повертає this (посилання на себе).
     Особливої уваги заслуговує метод Build(). Він виконує дві функції: віддає клієнту повністю готовий об'єкт SqlQuery і одразу ж створює новий, чистий екземпляр у полі _query. Завдяки цьому один і той самий об'єкт будівельника можна використовувати для генерації багатьох різних запитів без ризику змішати дані.
     Наостанок розглянемо, як ця архітектура використовується на практиці.
    64  // 4. Демонстрація (результат виконання)
    65  class Program
    66  {
    67      static void Main(string[] args)
    68      {
    69          var builder = new MySqlQueryBuilder();
    70
    71          // Використання Fluent API
    72          var query = builder
    73              .FromTable("Users")
    74              .SelectColumns("Id", "Name", "Email")
    75              .Where("Age > 18")
    76              .OrderBy("Name")
    77              .Build();
    78
    79          Console.WriteLine("Generated SQL Query:");
    80          Console.WriteLine(query.ToString());
    81      }
    82  }
     У клієнтському коді наочно видно всі переваги обраного підходу. Замість передачі десятка незрозумілих параметрів у звичайний конструктор класу, ми просто створюємо екземпляр будівельника і послідовно вказуємо, що саме нам потрібно. Такий код дуже легко читати, він самодокументований. Тільки після того, як всі необхідні кроки виконано, викликається метод Build(), який гарантує, що ми отримаємо правильно сконструйований об'єкт запиту.

     1  ﻿// 1. Продукт — складний об'єкт, який ми будуємо
     2  public class SqlQuery
     3  {
     4      public string Table { get; set; } = "";
     5      public List<string> Columns { get; set; } = new List<string>();
     6      public string WhereClause { get; set; } = "";
     7      public string OrderByClause { get; set; } = "";
     8      public override string ToString()
     9      {
    10          var cols = Columns.Any() ? string.Join(", ", Columns) : "*";
    11          var query = $"SELECT {cols} FROM {Table}";
    12
    13          if (!string.IsNullOrEmpty(WhereClause))
    14              query += $" WHERE {WhereClause}";
    15
    16          if (!string.IsNullOrEmpty(OrderByClause))
    17              query += $" ORDER BY {OrderByClause}";
    18
    19          return query + ";";
    20      }
    21  }
    22  // 2. Інтерфейс Будівельника
    23  public interface ISqlQueryBuilder
    24  {
    25      ISqlQueryBuilder SelectColumns(params string[] columns);
    26      ISqlQueryBuilder FromTable(string table);
    27      ISqlQueryBuilder Where(string condition);
    28      ISqlQueryBuilder OrderBy(string column);
    29      SqlQuery Build();
    30  }
    31
    32  // 3. Конкретний Будівельник
    33  public class MySqlQueryBuilder : ISqlQueryBuilder
    34  {
    35      private SqlQuery _query = new SqlQuery();
    36      public ISqlQueryBuilder SelectColumns(params string[] columns)
    37      {
    38          _query.Columns.AddRange(columns);
    39          return this; // Повертаємо current object для ланцюжка методів
    40      }
    41      public ISqlQueryBuilder FromTable(string table)
    42      {
    43          _query.Table = table;
    44          return this;
    45      }
    46      public ISqlQueryBuilder Where(string condition)
    47      {
    48          _query.WhereClause = condition;
    49          return this;
    50      }
    51      public ISqlQueryBuilder OrderBy(string column)
    52      {
    53          _query.OrderByClause = column;
    54          return this;
    55      }
    56      public SqlQuery Build()
    57      {
    58          var result = _query;
    59          _query = new SqlQuery(); // Скидання для наступного використання
    60          return result;
    61      }
    62  }
